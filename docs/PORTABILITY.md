# Портируемость IWE

> Читай этот файл если: не на macOS, не используешь Claude Code,
> или хочешь понять — нужно ли тебе менять конкретный раздел инструкций.

---

## Быстрый ответ: смотри на слой

IWE разделён на три слоя. Слой определяет, насколько правило привязано к платформе:

| Слой | Где | Платформенная зависимость |
|------|-----|--------------------------|
| **L1** — платформа | CLAUDE.md §1–7, `.claude/hooks/`, `.claude/skills/` | **Привязан к Claude Code и macOS** |
| **L2** — staging | CLAUDE.md §8 | Частично специфичен |
| **L3** — личное | `extensions/`, `params.yaml`, CLAUDE.md §9 | **Универсален — твоя логика** |

**Правило:** встретил инструкцию → посмотри, в каком слое она живёт. L3 менять не нужно. L1 — нужно при смене платформы.

## Kimi Code (VS Code)

**Работает из коробки — никакой дополнительной настройки не нужно.**

1. Установи расширение Kimi Code в VS Code.
2. Открой форкнутый репо — Kimi автоматически подхватит `AGENTS.md` из корня.
3. Работай как обычно: Kimi читает все инструкции из `AGENTS.md`.

**Кастомизация для Kimi:** клади свои инструкции в `extensions/` или в `AGENTS-agent-blocks.md`. Они попадут в `AGENTS.md` после запуска `bash scripts/sync-agent-instructions.sh`. Не редактируй `AGENTS.md` вручную — он перезапишется.

**Что не работает через Kimi:** скиллы (`/day-open`, `/ke`, `/archgate`) и хуки — это механизм Claude Code CLI. Для них нужен Claude Code.

## Codex

Codex может работать с IWE как проектный агент: он читает `AGENTS.md`, умеет редактировать файлы и запускать команды. Базовый сценарий не требует MCP или автоматизации.

### Базовый запуск

1. Установи Codex и открой корень репозитория.
2. Подтверди доверие к проекту. Только доверенный проект должен применять проектную конфигурацию из `.codex/config.toml`.
3. Проверь, что `AGENTS.md` актуален. Это сгенерированный файл: меняй источник и запускай `bash scripts/sync-agent-instructions.sh`, а не редактируй его вручную.
4. Запусти рабочий протокол вручную. Без MCP Codex работает с локальными файлами; недоступен только облачный слой знаний.

### Соответствие механизмов

| Задача | Claude Code | Codex |
|---|---|---|
| Постоянные инструкции | `CLAUDE.md` | `AGENTS.md` |
| Проектная конфигурация | `.claude/settings.json` | `.codex/config.toml` |
| Проектные скиллы | `.claude/skills/` | `.agents/skills/` |
| События жизненного цикла | `.claude/hooks/` | проектные hooks Codex и адаптеры событий |
| MCP | `.mcp.json` | MCP-серверы в `.codex/config.toml` |
| Неинтерактивный запуск | `claude -p` | `codex exec` |

Скиллы и hooks — не эксклюзивные механизмы Claude Code. Codex поддерживает собственные проектные скиллы и hooks, но форматы конфигурации, события и входные данные различаются. Поэтому содержимое `.claude/skills/` и `.claude/hooks/` нельзя считать автоматически совместимым: методологию протокола нужно вынести в общее ядро, а платформенные вызовы оформить тонкими адаптерами.

### Конфигурация, MCP и безопасность

- Проектная `.codex/config.toml` должна содержать только переносимые настройки. Секреты, токены и пользовательскую авторизацию в репозиторий не добавляй.
- Настройку `iwe-knowledge` для Codex следует генерировать из того же источника, что и `.mcp.json`, чтобы адреса и параметры не расходились.
- Если MCP не настроен, локальные Pack, DS, Base, memory и capture продолжают работать.
- Защитные hooks переносятся отдельно после сопоставления событий Claude Code и Codex. До проверки эквивалента ограничение должно оставаться явной инструкцией в `AGENTS.md` или проверкой обычного скрипта.

### Headless и расписание

Для автоматизаций используется общий backend с `IWE_AGENT_PROVIDER=claude|codex`. Codex запускается через `codex exec` с явным рабочим каталогом, `workspace-write`, тайм-аутом и файлом итогового ответа. Адаптер не включает `danger-full-access`, обход подтверждений или обход доверия к hooks. Расписание остаётся платформенной задачей: `launchd` на macOS, `systemd` timer или `cron` на Linux/WSL.

Все проектные skills синхронизируются из `.claude/skills/` в `.agents/skills/` с проверкой drift. Claude hooks проходят через Codex-адаптер событий; `SessionEnd`, которого нет в Codex, выполняется как fail-safe на `Stop`. `setup.sh` устанавливает Codex-контур, а `update.sh` обновляет платформенные файлы, сохраняя `USER-SPACE` и не перезаписывая существующие пользовательские `.codex/config.toml` и `.codex/hooks.json`.

## Hermes Agent

Два способа подключить Aisystant MCP — выбери удобный:

**Вариант А — через интерфейс:**
1. Settings → MCP Servers → Add → Aisystant.
2. Войди через Ory (откроется браузер).

**Вариант Б — через CLI:**
```bash
hermes mcp login aisystant
```
Если появляется ошибка OAuth — запусти `hermes mcp update aisystant` или проверь `hermes --help mcp`.

После подключения Hermes автоматически получит инструкции через инструмент `get_instructions`.

**Важно:** Hermes не читает файлы репо напрямую (в отличие от Claude Code и Kimi Code). Все инструкции он получает через MCP, а не из `CLAUDE.md` или `AGENTS.md`.

## Что зависит от Claude Code

Если ты используешь Cursor, Copilot, Gemini или другой ассистент (не Kimi и не Hermes) — следующие компоненты L1 требуют замены или адаптации:

- **Формат CLAUDE.md** — читается Claude Code при старте. Kimi читает `AGENTS.md` (уже в шаблоне). Другие ассистенты используют иные форматы (`.cursorrules`, системный промпт и т.д.)
- **`.claude/hooks/`** — конкретная схема событий Claude Code; для Codex нужны адаптеры и собственная конфигурация hooks
- **Содержимое `.claude/skills/` и слэш-команды** (`/day-open`, `/ke`, `/archgate`) — формат Claude Code; Codex-версии размещаются в `.agents/skills/`
- **MCP-серверы** — Hermes поддерживает. Другие ассистенты — зависит от реализации.
- **Выбор модели** (Opus / Sonnet / Haiku) — специфика Anthropic API

**Что переносится без изменений:** протоколы ОРЗ как концепции, Pack/DS/Base архитектура, логика memory и capture, всё в `extensions/` и `params.yaml`.

---

## Что зависит от macOS

Если ты на Linux или Windows (WSL) — следующие компоненты L1 требуют замены:

| Компонент | macOS | Linux / WSL |
|-----------|-------|-------------|
| Таймеры автоматизации | `launchd` + `.plist` | `systemd` timer или `cron` |
| Пробуждение ноутбука | `pmset repeat wakeorpoweron` | `rtcwake` / `WakeSystem=true` |
| Транскрипция `/transcribe` | MLX Whisper (Apple Silicon) | Требует замены модели |
| Пути | `~/Library/LaunchAgents`, Homebrew-пути | Стандартные Linux-пути |

`setup.sh` определяет ОС автоматически и пропускает macOS-шаги на Linux. Технические детали для контрибьюторов: [PLATFORM-COMPAT.md](PLATFORM-COMPAT.md).

---

## Что универсально (не трогай)

Независимо от платформы и ИИ-ассистента — следующее работает без изменений:

- **ОРЗ-фрактал** (Открытие → Работа → Закрытие) — принцип, не инструмент
- **Pack / DS / Base архитектура** — организация знаний в файлах
- **Memory и Capture** — логика фиксации знаний
- **Протоколы** (Day Open, Week Close, Strategy Session) — как концепции
- **Весь L3:** `extensions/`, `params.yaml`, личные различения

---

## Как адаптировать под другую платформу

1. Перенеси L3 (`extensions/`, `params.yaml`) без изменений — это твоя логика
2. Перепиши L1 под формат нового ассистента (системный промпт, `.cursorrules` и т.д.)
3. Концепции протоколов (ОРЗ, Pack, capture) — опиши в новом формате: они платформенно-нейтральны

## Добавить поддержку нового агента

> Для разработчиков, которые хотят интегрировать агент, которого здесь ещё нет.

IWE не привязан к конкретному LLM или агенту. Любой агент, удовлетворяющий минимальному контракту, будет работать с IWE без изменений в Pack, протоколах и L3.

### Минимальный контракт

| Требование | Зачем | Уровень |
|---|---|---|
| Читает файл инструкций из проекта | CLAUDE.md / AGENTS.md / системный промпт | **Обязательно** |
| Tool use: редактирует файлы, запускает команды | правки кода, скрипты протоколов | **Обязательно** |
| Поддержка MCP | Aisystant cloud: знания, персона, Digital Twin | Опционально |
| Headless / CLI режим | автоматизации (cron, хуки) | Опционально |

Без MCP агент работает автономно — теряется только облачный слой знаний.
Без headless-режима — ручной запуск вместо расписания.

### Как добавить адаптер

1. **Создай файл инструкций** — адаптер для чтения агентом. Примеры:
   - Claude Code читает `CLAUDE.md`
   - Kimi Code читает `AGENTS.md` (генерируется из CLAUDE.md скриптом `sync-agent-instructions.sh`)
   - Hermes получает инструкции через MCP `get_instructions`
   - Cursor — `.cursorrules`, Copilot — системный промпт через `.github/copilot-instructions.md`

2. **Добавь агент в `setup.sh`** — включи имя CLI-команды в `AI_CLI_CANDIDATES` (через env или хардкод в скрипте).

3. **Задокументируй** — добавь раздел в этот файл (PORTABILITY.md) по аналогии с Kimi Code и Hermes Agent выше.

4. **Pull request приветствуется** — если хочешь добавить агент для всех пользователей IWE.

### Политика изменений в форке и upstream

Поддержка нового агента сначала реализуется и проверяется в форке: upstream-ревью не блокирует работу форка. Для каждой законченной порции изменений:

1. Создай upstream issue с проблемой, ожидаемым поведением и результатами проверки.
2. Если решение универсально, подготовь небольшой отдельный pull request в upstream.
3. Свяжи локальный коммит, issue и pull request в описании изменений.
4. Оставь локальные настройки, пути, секреты и персональную логику только в форке.
5. Укажи, как сохраняется совместимость с Claude Code и другими уже поддержанными агентами.

Предпочтительны небольшие независимые изменения: документация, генерация инструкций, обнаружение CLI в setup, скиллы, hooks, MCP/headless и тесты совместимости — отдельными issue и pull request.
