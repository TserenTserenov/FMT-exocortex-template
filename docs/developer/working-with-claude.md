# Работа с Claude в разработке

> Для разработчиков T4+ (ведущие разработчики). Claude Code — экзоскелет для мышления, а не автопилот. В этом документе — практика из `memory/reference/agent-core.md` и реальных сессий разработки IWE.

## Когда Claude полезен / когда опасен

### ✅ Полезен

1. **Исследование неизвестного** — новая библиотека, API, инструмент. Claude быстро соберёт паттерны из документации. **Правило:** всегда проверяй примеры на реальных версиях, которые ты используешь.

2. **Рефакторинг по спецификации** — когда тесты уже написаны (станция 4, [testing-as-spec.md](testing-as-spec.md)), а код грязный. Claude предложит несколько вариантов, ты выбираешь, запускаешь тесты.

3. **Boilerplate и шаблоны** — конфиги, типовые обработчики ошибок, структуры данных. Экономит механическую работу, если спец-логики мало.

4. **Копирование и адаптация из соседних файлов** — Claude помнит локальный стиль из `.claude/rules/distinctions.md` и вашего кода, адаптирует паттерн.

5. **Документирование известного кода** — Claude придумает примеры для docstring, если логика кода чётко определена.

6. **Peer-диалог на сложном дизайне** — обсудить идею перед кодом: вероятных ошибок меньше. Ты думаешь вслух, Claude ловит противоречия. **Выполняешь ты сам**, диалог ускоряет, но ответственность твоя.

### ❌ Опасен

1. **Архитектурные решения без ArchGate** — Claude может предложить элегантное решение, которое сломает интеграцию с платформой. **Обязательно пройди ArchGate первым** (CLAUDE.md §5).

2. **Copy-paste без локализации** — паттерны из интернета или соседних проектов, но в контексте IWE несовместимы (версии, permissions, API, хуки). **Всегда адаптируй, не копируй.**

3. **Доверие "consensus"** — две сессии Claude согласились на решение ≠ проверено. Нужна независимая валидация (live run, code review, тест) перед merge. Даже консенсус двух агентов требует третейского судьи.

4. **Галлюцинации на API** — Claude может уверенно написать вызов функции, которой нет в текущей версии. **После генерации → сразу `pytest`; ошибка → диагностируй вместе.**

5. **Логирование PII** — Claude может захватить пользовательские данные в логи/трассировку. **Security Gate (CLAUDE.md pre-action gates) обязателен перед реализацией**, если РП касается персональных данных.

## Workflow: от WP Gate к merge

### Шаг 0: WP Gate (перед обращением к Claude)

**БЛОКИРУЮЩЕЕ.** Читай `memory/protocol-open.md` ДО начала. Объяви:
- **Роль:** что ты играешь (разработчик, архитектор, диагност)?
- **Работа:** конкретно что (спецификация, дизайн, реализация)?
- **Класс верификации:** как проверится (closed-loop = тесты; open-loop = peer-сессия)?
- **Оценка и модель:** время, LLM-модель если нужна.

**Пример:**
```
Пилот: Напиши обработчик ошибок для API-запросов.

Claude: ❌ Недостаточно. Вопросы:
- Новая функция или рефакторинг существующей?
- На каких ошибках фокусироваться (5xx, timeout, malformed)?
- Есть ли логирование? Какое?
- Класс верификации? (тесты готовы?)

Пилот: Новая. 5xx ошибки. Логирование готово. Тесты готовы.
Класс: closed-loop (есть тесты).

Claude: ✅ WP-XYZ открыта. Прочитал тесты, вижу спец.
```

### Шаг 1: Design-talk (если нетривиально)

Если рутинно (тест → код) → skip. Если спорно (два варианта решения):
- Спроси Claude: какие подходы? trade-offs?
- **Ты выбираешь**, какой лучше для вашего проекта.
- Проговори опасности: где может сломаться?

**Пример:**
```
Claude: Два подхода:
  A) Retry с экспоненциальной задержкой (просто, но может зависнуть).
  B) Circuit Breaker (сложнее, но масштабируется).

Вы масштабируете на тыс запросов/мин?

Пилот: Нет. Хватит A.

Claude: OK. А риск: конфликт таймаута в LB и нашего retry.
Какой таймаут в вашем балансировщике?
```

### Шаг 2: Реализация (code-pair)

**Prompt-паттерны:**

**«Объясни сначала»** — для нетривиального (новая функция, побочные эффекты, граница модулей). План в 3–5 пунктов до кода. Дешевле поймать ошибку в плане, чем в 200 строках.

**«Сразу пиши»** — для тривиального (тест по спеке, однострочный фикс). Промежуточное объяснение здесь — трата времени.

**Примеры промптов:**
```
Напиши функцию validate_user(data: dict) -> Tuple[bool, str].
Требования:
- Проверить type_id в диапазоне [1..999]
- Проверить email на формат SMTP
- На ошибке вернуть False + текст ошибки
Примеры:
  {"type_id": 42, "email": "test@ex.com"} → (True, "")
  {"type_id": 0, "email": "bad"} → (False, "invalid type_id")
Стиль: как в src/validators.py (повтори импорты, именование)
```

**ПРАВИЛО: Запусти тесты сразу после генерации.** Не жди.

```bash
pytest tests/test_validators.py::test_validate_user -v
```

Ошибка? Диагностируй вместе с Claude. Регрессия в соседних тестах? Сразу видно.

### Шаг 3: Проверка перед commit

**Чеклист:**

- [ ] **Тесты прошли полностью.** `pytest` (весь suite), не выборочно.
- [ ] **Нет дублирования кода (P2).** Если видишь повторение блока в 3м месте — рефактори общую функцию, не копируй.
- [ ] **Нет мёртвых веток (P3).** Удали код, который никто не вызывает, не оставляй «на будущее» без потребителя.
- [ ] **Нет `except: pass` без логирования (P4).** Хотя бы логируй или выпусти исключение явно.
- [ ] **Нет логирования PII.** Grep по именам полей (email, phone, personal_id, secret). Если логируешь context — проверь, что там не попадут данные.
- [ ] **Долгая функция разбита (P5).** Функция делает несколько дел (парсинг + валидация + запись) → разбей на части.

Форматирование (P0) проверяет pre-commit hook.

**Git staging (КРИТИЧНО):**

```bash
# ❌ ЗАПРЕЩЕНО — подхватит чужую работу (parallel agents)
git add -u
git add .
git add -A

# ✅ ВСЕГДА — только конкретные файлы
git add src/handlers/api.py tests/test_handlers.py

# Перед коммитом — проверить scope
git diff --cached --name-only
# Если появились неожиданные файлы → git restore --staged <file>
```

**Commit message:**
```
Fix: retry logic for API errors

- Add exponential backoff (1s, 2s, 4s, 8s max)
- Log retry attempts for debugging
- Preserve original error in fallback

Fixes WP-XYZ stage #3

Co-Authored-By: Claude Haiku <noreply@anthropic.com>
```

### Шаг 4: Peer-review (ведущий разработчик)

Ведущий разработчик (TD1+TA4) читает:
- Логика ≟ спецификация РП?
- Тесты покрывают граничные случаи?
- Стиль (code-style.md)?
- Нет регрессий в соседних компонентах?

Вопросы → вернуть в разработку (шаг 2), не merge.

## Capture (двойной выход)

На рубеже, когда код готов → **что нужно задокументировать?**

**Правило:** задача без capture = **незакрытая**. Code alone isn't enough.

| Что сделал | Capture → |
|-----------|-----------|
| Новый паттерн retry-logic | `memory/lessons_retry_patterns.md` — когда использовать, trade-offs |
| Нашёл подводный камень (race condition в concurrent write) | `docs/reference_db_concurrency.md` — как мы это решили |
| Интегрировал новую библиотеку | `docs/dependencies-changelog.md` + `CONTRIBUTING.md` (версии) |
| Нестандартное решение по security | `docs/security-decisions.md` + ArchGate-профиль в архив |

**Анонс перед фиксацией:**
```
Capture: Race condition в concurrent updates → docs/reference_db_concurrency.md
Три разработчика по-разному решали, как миксировать UPDATE+SELECT в одной транзакции.
Фиксируем решение, чтобы не повторили ошибку.
```

## Частые ошибки (как избежать)

| Ошибка | Причина | Fix |
|-------|--------|-----|
| Claude сгенерировал код со старым API | версия библиотеки в контексте устарелая | Перед запросом: «Я использую `requests` v2.31. Покажи пример.» |
| Два варианта решения — выбрал не тот | пропущен ArchGate | **ВСЕГДА ArchGate перед реализацией** (CLAUDE.md §5) |
| Тесты не запустил → merge → регрессия | спешка | После генерации → терминал → `pytest` → потом читаешь результат |
| Скопировал код из интернета, не адаптировал | гнал пулл-реквест быстрее | Grep по импортам (ваша версия?). Run пример локально. |
| PII попало в логи | Claude не знал о sensitivity | Security Gate (CLAUDE.md) — перед работой с personal data |
| Третье повторение вместо функции (P2) | Claude копирует, не факторит | Попроси явно: «вынеси общую логику в функцию» |
| Мёртвая ветка кода (P3) | Claude оставляет «на будущее» | Удали без TODO — если никто не вызывает → вон |

## Пример хорошей сессии (closed-loop, простой случай)

```
[WP-452 Ф2] Написать валидатор для user_input

Claude: Прочитал тесты. Уточняющие вопросы:
  - Только синтаксис email (SMTP)?
  - Или проверка MX-рекорда домена?

Пилот: Только синтаксис. Дальше не нужно.

Claude: [Генерирует validate_user_input + примеры]

Пилот: [Запускает pytest в терминале]
  ✅ 12 passed, 0 failed

Claude: ✅ Готово. Что стоит зафиксировать?

Пилот: Ничего специального. Стандартный валидатор.

[Commit] Fix: validate_user_input for WP-452 (Ф2 done)
```

**Что сработало:**
1. WP Gate ДО кода (класс верификации — closed-loop).
2. Claude спросил уточняющий вопрос (не угадал).
3. Код проверен на тестах сразу.
4. Capture не требовался (стандартное решение).

## Пример опасной сессии (не сделать так)

```
[WP-500] Переписать database schema

Claude: Текущая схема 3NF. Предлагаю денормализацию для скорости чтения.

Пилот: Звучит логично. Делай.

❌ ОШИБОк: 
- Нет ArchGate (архитектурное решение).
- Нет peer-сессии перед кодом.
- Нет проверки: вы правда часто читаете Reports?
```

**Как избежать:**
- ArchGate ПЕРВЫМ для архитектурных решений.
- open-loop класс → peer-сессия перед кодом, не после.
